1. Идея проекта и его цель (вход в 5 мин. демо по UI)
----------------------------------------------------

CI/CD Monitor — инструмент на Python с веб-дашбордом, который собирает в одном месте:

  • статусы сборок и пайплайнов (несколько экземпляров Jenkins и GitLab в конфиге);
  • сводку по тестам (в т.ч. из отчётов pytest/Allure при сборе);
  • состояние контейнеров и HTTP-проверок (Docker monitor);
  • при необходимости — отчёты (HTML/CSV), уведомления в Telegram (несколько ботов), история в SQLite.

Данные периодически подтягиваются в снимок (Collect / авто-сбор в настройках); в интерфейсе видно
актуальное состояние без прыжков по разным UI CI.

Почему сделали (идея)

  • много пайплайнов в разных системах;
  • много тестов — неочевидно, что «горит» в первую очередь;
  • рядом контейнеры и сервисы — связь «упал билд» / «лег стенд» не всегда очевидна.

Идея: одна точка «как сейчас CI + тесты + инфра», с логами, фильтрами, трендами и при желании
ИИ-ассистентом по контексту дашборда.

Что сказать в одном абзаце (готовая формулировка)

«Мы не заменяем Jenkins/GitLab — агрегируем их в снимок и даём слой мониторинга: что упало, что
критично, что с сервисами, куда кликнуть за логом, как смотреть динамику и инцидент — в одном
дашборде, плюс настройки через веб без ручного редактирования YAML для типовых вещей.»


2. Демо по UI: кнопки и вкладки (5 мин.: что показываем и что говорим)
----------------------------------------------------------------------

Идём слева направо по шапке, затем по вкладкам. На каждом шаге: что нажать/показать и какой смысл для аудитории.

2.1. Шапка и глобальные элементы
--------------------------------

Показать по порядку (верхняя полоса):

  Collect
      Принудительно запустить сбор данных с источников (Jenkins/GitLab/Docker и т.д.) и обновить
      снимок. Показать строку статуса сбора (идёт / готово / ошибка).
      В коде: `web/app.py` → `trigger_collect()` (`POST /api/collect`) и `collect_status()` (`GET /api/collect/status`).
      Важно: `POST /api/collect` защищён shared-token’ом, если он задан (см. ниже про `X-API-Token`).

  FULL (переключатель рядом с Collect)
      Форсирует полный рескан (игнорировать инкрементальные кеши/оптимизации) — полезно, если данные
      “залипли” или поменялся список джоб/проектов.
      В коде: `web/app.py` → `trigger_collect()` читает `{"force_full": true}`.

  Refresh
      Перечитать таблицы из уже сохранённого снимка на диске, без полного collect.
      Важно: это клиентская перезагрузка данных, без запуска сборщика на сервере.
      В коде: UI функция `refreshAll()` в `web/templates/index.html`.

  LIVE (переключатель)
      Включает фоновый опрос на клиенте (статус сбора, уведомления) и авто-обновление UI.
      Также LIVE включает/выключает серверный auto-collect через `POST /api/collect/auto` (то есть влияет
      на то, будет ли сервер сам запускать сбор по интервалу).
      Важно: `POST /api/collect/auto` защищён shared-token’ом, если он задан (см. ниже про `X-API-Token`).

  Тема / язык (EN | RU) / компактный режим
      Удобство чтения и плотность таблиц.

  CSV
      Экспорт CSV по сборкам из вкладки Builds с учётом фильтров (source/instance/status/job) —
      выгрузка делается на клиенте (UI запрашивает `/api/builds` большим лимитом и формирует файл).
      Дополнительно в UI есть ссылки на серверный экспорт:
      `GET /api/export/builds`, `GET /api/export/tests`, `GET /api/export/failures`.

  Incident ▾
      Быстрый доступ к экспорту снимка инцидента: JSON и Markdown (удобно вставить в тикет).
      Реальные URL: `/api/incident.json` и `/api/incident.md` (есть также алиасы `/api/export/incident*`).
      В коде: `web/app.py` → `_export_incident_response()` и роуты `/api/incident.*`.

  Настройки
      Отдельная страница: несколько Jenkins/GitLab, Telegram-боты, Docker/HTTP, авто-сбор,
      ИИ (OpenAI, Gemini, OpenRouter, Cursor proxy, Ollama локально и др.).

  Поиск (глобальный)
      Фильтрация уже загруженных строк на текущих вкладках (не запрос к CI).

  Колокол уведомлений
      Лента событий смены состояний (после появления данных).

Плавающая кнопка ИИ (если включена конфигурация)
      Чат с моделью по контексту дашборда; провайдер задаётся в Настройках → ИИ (в т.ч. Ollama
      на http://127.0.0.1:11434/v1 и модель вроде llama3.1:8b).
      В коде: `web/app.py` → `/api/chat` и `/api/chat/status`.

Глубокие ссылки на вкладки
      Можно открыть сразу нужный раздел, например:  ?tab=trends  или  ?tab=incidents
      (остальные: overview, builds, tests, services, logs).
      В коде: `web/templates/index.html` → чтение `new URLSearchParams(location.search).get('tab')`.


2.2. Обход по вкладкам (как на панели табов)
--------------------------------------------

Ниже порядок: Обзор → Сборки → Тесты → Сервисы → Тренды → Инциденты → Logs.
На каждой вкладке: что показать руками и что сказать вслух.


2.2.1. Обзор (Overview)
-----------------------

Подзаголовок в UI: метрики, карта, ситуация.

Что показать

  • Карточки метрик: всего сборок, успешные / упавшие / в процессе, упавшие/пройденные тесты,
    сервисы down, flaky — клик по части карточек ведёт на отфильтрованные Сборки/Тесты или к блоку
    flaky на Сервисах.
  • Легенда цветов статусов.
  • Полоса ситуации (если есть проблемы) — куда копать дальше.
  • Карта статусов (если включена и есть данные) — теплокарта; клик по ячейке может отфильтровать
    сборки.
  • При наличии — блок инцидентов/сводки на обзоре, аномалии, корреляции.

Фраза для аудитории

«Это пульт: за секунды видно, зелёное всё или есть красные зоны, и куда идти — в сборки, тесты
или сервисы.»


2.2.2. Сборки (Builds)
----------------------

Подзаголовок: пайплайны, фильтры, избранное.

Что показать

  • Избранное (Starred) — закреплённые джобы/проекты.
  • Таблица: источник, джоб, номер, статус, контекст (streak / последний OK), ветка, время,
    ссылки, лог, действия (запуск пайплайна — если настроено и разрешено).
  • Фильтры: источник, статус, текст по джобу, окно времени (24h / 7d и т.д.), пресеты вроде
    «падения за 24ч» / «избранное».
  • При большом объёме — подгрузка при прокрутке.

Фраза

«Операционный слой CI: кто упал, когда, на какой ветке; отсюда лог и при необходимости перезапуск.»


2.2.3. Тесты (Tests)
--------------------

Подзаголовок: топ падений и все прогоны.

Что показать

  • Топ падающих тестов — агрегат, фильтры по имени/suite, экспорт (CSV/XLSX — по доступным
    кнопкам в вашей сборке).
  • Все прогоны — построчно: тест, suite, статус, длительность, время, сообщение об ошибке.

Фраза

«Отдельно от статуса билда — качество тестов: что стабильно красное, что флакует; удобно QA
и разработке.»


2.2.4. Сервисы (Services)
-------------------------

Подзаголовок: контейнеры, flaky, лента.

Что показать

  • Таблица сервисов/контейнеров: имя, тип, статус, деталь, время проверки, uptime при наличии
    истории в БД.
  • Flaky jobs (если есть данные) — джобы с чередованием успех/падение.
  • Лента событий (timeline) — переходы состояний и недавняя активность.

Фраза

«То, что не всегда видно из одного билда: контейнер down, HTTP красный — рядом с CI. Отсюда же
логи Docker и действия start/stop/restart, если это разрешено конфигом и безопасно для среды.»


2.2.5. Тренды (Trends)
----------------------

Подзаголовок: графики по времени.

Что показать

  • Пресеты периода: 3 / 7 / 14 / 21 / 30 дней (данные накапливаются по снимкам; длинный запрос
    подгружает до ~30 дней, срез можно менять на клиенте).
  • Размер графиков: S / M / L / XL; полноэкран по кнопке на карточке графика.
  • Панель фильтров:
      — сглаживание линий стандартных графиков (выкл. / скользящее 3 или 7 дней);
      — Top N для столбчатого «топ падающих тестов»;
      — опционально диапазон дат «с — по» + Применить / Сброс.
  • Стандартные графики: сборки (всего/упавшие), тесты, сервисы down, топ падающих тестов.
  • «Новый график…» — конструктор своих графиков: линии/столбцы, метрики по дням, в т.ч. доли
    падений, привязка к конкретному job или сервису из данных, сохранение в localStorage.

Фраза

«Не только „сейчас“, но и динамика: растёт ли доля падений, не деградирует ли качество за неделю;
можно собрать свой график под команду.»


2.2.6. Инциденты (Incidents)
----------------------------

Подзаголовок: сводка и коррелированные падения.

Что показать

  • Центр инцидентов (когда есть что показывать): уровень серьёзности, мета, затронутые джобы/
    сервисы, таймлайн.
  • Экспорт JSON/Markdown — с вкладки или из шапки (Incident).
  • Коррелированные сбои (если есть) — несколько падений в окне; переход к фильтрам сборок.

Фраза

«Когда падает несколько вещей сразу — одна картина для эскалации и ретро, плюс готовый текст
в тикет.»


2.2.7. Logs (Collect / parsers)
-------------------------------

Подзаголовок: живые логи сбора и “топ медленных” операций.

Что показать

  • Логи сбора (Collect logs): фильтры по уровню, инстансу, фазе, job, текстовый фильтр.
  • Топ медленных операций (slow) — помогает быстро объяснить “почему Collect долго”.
  • Полезный сценарий: нажать Collect → перейти на Logs → увидеть ретраи/429/503 и понять,
    где узкое место.

В коде

  • backend: `web/app.py` → `collect_logs()` (`GET /api/collect/logs`) и `collect_slow()` (`GET /api/collect/slow`)
  • UI: `web/templates/index.html` → `loadCollectLogs()`, `loadCollectSlowTop()`


2.3. Настройки (страница /settings)
-----------------------------------

Кратко упомянуть в конце демо или при вопросе «как подключили Jenkins».

  • Несколько экземпляров Jenkins и GitLab — каждый со своим URL и токеном, кнопки добавить/
    удалить карточку.
  • Уведомления: несколько Telegram-ботов (токен, chat id, опционально свой API base для
    self-hosted Bot API).
  • Общее: авто-сбор, интервал, глубина lookback, топ N тестов в отчётах.
  • Docker / HTTP: списки контейнеров и URL проверок.
  • ИИ: провайдер (в т.ч. Ollama локально), модель, base URL, прокси при необходимости.

После сохранения конфиг пишется в config.yaml, цикл сбора перезапускается.
В коде: `web/app.py` → `api_save_settings()` (`POST /api/settings`).

Shared token (опционально, про безопасность)
-------------------------------------------

В `web/app.py` есть защита “опасных” эндпоинтов общим токеном. Если он задан, запросы должны иметь
заголовок `X-API-Token: <token>` (или `Authorization: Bearer <token>`). Источники токена:

  • env: `CICD_MON_API_TOKEN`
  • config: `web.api_token`

Эндпоинты под токеном включают как минимум: `POST /api/collect`, `POST /api/collect/auto`,
`GET/POST /api/settings`, действия (`/api/action/*`), просмотр логов (`/api/logs/*`) и чат (`/api/chat*`).
Если токен включён, но прокси/браузер его не подставляет — соответствующие кнопки в UI будут получать 401.

4. Разработка с Cursor (рассказ, ~4 мин.)
-----------------------------------------

О чём говорить: как шёл процесс с ИИ в Cursor, не пересказ кода.

  • С чего начали: постановка цели, первый скелет (FastAPI, шаблоны, сбор данных).
  • Как работали с агентом: типичные запросы (фича → правки в app.py / index.html), итерации,
    когда переписывали промпт или разбивали задачу на шаги.
  • Что ускорило: автодополнение, поиск по репо, генерация роутов/таблиц, разбор ошибок из логов.
  • Где смотрели сами: диффы, запуск локально, тесты/ручная проверка UI — «ИИ предлагает, человек
    проверяет».
  • Один пример «до/после» или самого полезного совета по работе в Cursor (по желанию).


5. Чему научился(лась) (~1 мин.)
--------------------------------

Коротко, 1–3 тезиса — про разработку в паре с ИИ (Cursor), не про синтаксис языка.

  • Формулировать задачу так, чтобы агенту хватало контекста: цель, ограничения, файлы/эндпоинты,
    ожидаемое поведение — тогда меньше «красивого мусора» и переделок.
  • Смотреть дифф и прогонять сценарий руками: ИИ ускоряет набросок и рутину, ответственность за
    корректность и безопасность остаётся за человеком.
  • Дробить большую фичу на шаги и уточнять промпт после первой версии — быстрее, чем один
    огромный запрос «сделай всё».
  • Использовать ИИ как «вторую голову» для разбора логов и навигации по большому репозиторию,
    но опираться на официальные доки и реальный вывод приложения, когда поведение критично.

(Оставьте в выступлении любые 2–3 пункта, которые для вас правда.)


Приложение: возможные вопросы (коротко)
--------------------------------------

  Это замена Jenkins?
    Нет, надстройка: читает API и показывает сводку.

  Данные в реальном времени?
    Снимок обновляется по Collect и авто-интервалу; LIVE ускоряет опрос статуса на клиенте.

  Где хранится история?
    Снимок — файлы в data/; при включённом SQLite — история сборок/событий для графиков и таймлайнов.

  Безопасно ли рестартить Docker из UI?
    Зависит от прав и конфига; в проде обычно ограничивают список контейнеров и среду.

  Как подключить локальную Ollama?
    Настройки → ИИ → провайдер Ollama, base URL http://127.0.0.1:11434/v1, модель как в
    `ollama list` (например llama3.1:8b), ключ чаще всего не нужен.
